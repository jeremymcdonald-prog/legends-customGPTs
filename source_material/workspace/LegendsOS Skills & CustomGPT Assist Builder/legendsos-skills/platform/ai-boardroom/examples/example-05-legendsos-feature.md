# Example 05 — LegendsOS Product Feature Review

## Input

> LegendsOS feature review: an Atlas button that drafts a borrower update email from the loan file and sends it.

## Output

**1. EXECUTIVE VERDICT**
Fix-then-ship. The drafting half is a strong feature. The sending half cannot ship as a one-click action. Split the feature into draft + approve, and this becomes one of the highest-leverage moves Atlas can ship this quarter.

**2. BOARDROOM WORKFLOW SELECTED**
Workflow 13 — LegendsOS Product Feature Review. Operational + Strategy. Architect, Data, Risk, CTO, Compliance, Offer, Customer/Borrower active.

**3. ADVISOR SEAT REVIEWS**
- **Architect Seat:** the loan file lives in TERA. Atlas should read from TERA, not duplicate borrower state. If Atlas builds its own loan-file mirror, R2 fires.
- **Data Seat:** this is borrower-adjacent text by definition. Use Claude in-stack. PII redaction is not optional. Loan numbers, addresses, dollar amounts must be classified separately from the LLM step.
- **Risk Seat:** Tier 3 the moment "send" is part of the button. Tier 2 if it's "draft and route to the LO's inbox for review and send."
- **CTO Seat (would Thuan ship this?):** the draft-and-route version, yes, with the right audit. The one-click send version, no. It's the kind of feature that looks brilliant on demo day and creates a real incident in week three.
- **Compliance Seat:** any external borrower email needs NMLS + EHO, accurate sender identity, and no rate / approval claims that weren't in the file. The keyword filter has to fire. AI-written external bodies stay in draft for human approval (90-day minimum) per the standing rule.
- **Offer Seat:** the value is real. Borrowers want updates. LOs hate writing them. A clean draft saves 15 to 20 minutes per file per week.
- **Customer/Borrower Seat:** the borrower needs to feel they're hearing from their LO, not a system. Voice and "Sent on behalf of [LO]" are essential.

**4. POINTS OF AGREEMENT**
Every seat agrees the drafting half is high-value. Every seat agrees the one-click send is wrong.

**5. POINTS OF CONFLICT**
Speed (Offer) vs Send-approval (Compliance / Risk). Resolution: split the feature. Atlas drafts. LO approves and sends from their own account. The audit log records the full chain.

**6. HIDDEN RISKS**
- **Tier 3 Red** if "send" stays on the button. **Tier 2 Yellow** in the draft-and-route version.
- **R4** if the body goes external without LO review.
- **R10** if any path auto-sends.
- **R7** if Atlas drops "sent-on-behalf-of" or NMLS in the draft and LO sends as-is without noticing.
- GLBA exposure if borrower PII lands in any logging or telemetry without redaction.

**7. BEST OPPORTUNITY**
This becomes a daily Atlas action. Every active loan generates a suggested update on a cadence Atlas knows from the file (e.g., new condition logged → suggested borrower update drafted). The leverage compounds across the team.

**8. RECOMMENDED DECISION**
Fix-then-ship.

**9. PRIORITY FIXES (ranked)**
1. Drop "send" from the button. Replace with "Draft borrower update." Output lands in the LO's Atlas inbox preview, then in their email draft folder when they hit Approve.
2. Atlas reads borrower context from TERA. No mirrored loan-file state inside Atlas.
3. PII redaction runs before any Claude call. Loan numbers and dollar amounts are classified deterministically and re-inserted post-draft, not generated by the model.
4. Drafts default to LO voice + "Sent on behalf of [LO]" footer + NMLS #1195266 (or the relevant LO) + Loan Factory NMLS #320841 + EHO. None of those fields are LO-editable in the draft.
5. Keyword filter fires before draft is offered: CFPB, RESPA, TILA, fair lending, discrimination, harassment, retaliation, lawsuit, attorney, predatory, OFAC. On hit, no draft is generated; route to Compliance.
6. Audit log every draft, approval, and send with `request_id`, `org_id`, `loan_id`, `lo_id`, draft body hash, sent body hash, timestamp.
7. 90-day human-approval window minimum. Reassess automation only after the failure modes are visible.

**10. FINAL ACTION PLAN**
- Today: rescope the spec in LegendsOS to "Draft borrower update" only.
- This week: define the TERA read interface, the PII redaction step, and the audit log schema.
- Build: ship the draft-and-route version behind a feature flag for Jeremy's loan pipeline only. Measure time saved per file. Watch the failure log.
- After 4 weeks of clean data: open it to the next 3 LOs.
