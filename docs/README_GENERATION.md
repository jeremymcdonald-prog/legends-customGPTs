# README generation

## What generates `README.md`

The root `README.md` is generated by `scripts/generate_readme.rb`. The generator reads approved repository metadata, validates its configuration, derives current counts and status tables, and writes one deterministic Markdown document.

The generated UTC timestamp is anchored to the current release date in `CHANGELOG.md`. This keeps the published release provenance visible without creating a new diff every time the generator runs.

## Sources of truth

- `VERSION` and `CHANGELOG.md` — current semantic version and release date.
- `config/readme_sections.yaml` — title, mission, section order, repository map, documentation links, badges, scan sources, exclusions, sensitive-path patterns, portfolio limit, timestamp behavior, and status order.
- `core/` — architecture, audiences, compliance, CTA, routing, lead-capture tests, package structure, identity formatting, and other canonical modules.
- `config/profiles/` — approved professional identity used to render the compliance disclosure.
- `gpts/*/manifest.md` — generated package names, versions, and lifecycle statuses.
- `exporters/platforms.yaml` — platform targets, adapter status, and Action format.
- `docs/reports/TOP_15_CUSTOM_GPTS_REPORT.md` — ranked portfolio names, scores, and current first-five recommendations.
- `LICENSE_STATUS.md` — current licensing boundary.
- Git metadata — branch and public repository link when Git is available.
- The configured archive path — regular-file count only; the generator never reads or publishes archived source contents or filenames.

## Regenerate the README

From the repository root:

```bash
ruby scripts/generate_readme.rb
```

To verify freshness without modifying the file:

```bash
ruby scripts/generate_readme.rb --check
```

The script also supports `--config PATH` and `--output PATH`. Both paths must resolve inside this repository.

## Change the configuration

Edit `config/readme_sections.yaml`, then regenerate. Keep configured paths repository-relative and point them only at approved metadata. The generator fails with a clear error when a required key, section, source, status, pattern, or linked document is missing or invalid.

## Dynamic sections

The current release, release date, branch, repository URL, badges, architecture flow, statistics, audiences, export targets, repository map, package standard, compliance disclosure, Top 15 portfolio, first-five recommendations, lifecycle status tables, documentation links, license status, and provenance footer are derived at generation time.

## Files that must not be edited directly

Do not hand-edit the root `README.md`. Change its canonical source or `config/readme_sections.yaml`, then regenerate it. Generated GPT package snapshots are governed separately by the Factory builders and likewise must not be hand-edited.

## Future CI enforcement

A future CI workflow can install Ruby, run the repository validation suite, and enforce README freshness with:

```bash
ruby scripts/generate_readme.rb --check
```

No CI badge is published until such a workflow actually exists.

## Troubleshooting

- `Missing required README config key` or `Missing required README section`: restore the named key or section in `config/readme_sections.yaml`.
- `README source does not exist`: correct the configured relative path or restore the source file.
- `Documentation link does not resolve`: repair the configured local documentation path.
- `README.md is stale`: run the generator and commit the resulting README with the source change.
- Unexpected count or table: inspect the configured statistics source, manifest front matter, audience/export YAML, or Top 15 report rather than editing the rendered value.
- Git metadata unavailable: the generator falls back to explicit local values and still produces deterministic output.

## Acceptance criteria

1. `ruby scripts/generate_readme.rb` exits successfully.
2. A second run with unchanged inputs produces no diff.
3. `ruby scripts/generate_readme.rb --check` passes.
4. Dynamic counts and tables match their configured sources.
5. Every local Markdown link resolves.
6. The README contains no secret, credential, borrower-identifiable information, hidden environment value, or raw archived source content.
7. The existing Factory validation and test suites pass.
